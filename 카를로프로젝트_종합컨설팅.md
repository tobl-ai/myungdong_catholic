## 2. 현행 시스템 진단

### 2.1 굿뉴스 (catholic.or.kr) - 현황 분석

**기술 스택 노후화 수준: 극심**

| 항목 | 현재 상태 | 업계 표준 (2026) | 기술 격차 |
|------|----------|----------------|---------|
| 백엔드 | **Classic ASP** (1996년 기술) | Next.js / Node.js / Python | **30년** |
| 프론트엔드 | jQuery + 바닐라 JS | React / Vue / Svelte | 15년+ |
| 레이아웃 | **테이블 + 프레임셋** | Flexbox / Grid + 반응형 | 20년+ |
| API | XML over AJAX | REST / GraphQL + JSON | 15년+ |
| 모바일 | 별도 모바일 사이트 | 반응형 단일 코드베이스 / PWA | 10년+ |
| 인증 | 쿠키 기반 세션 | OAuth 2.0 / JWT / Firebase Auth | 10년+ |
| 호스팅 | **On-Premise** | Cloud (GCP/AWS) | 10년+ |
| AI | 없음 | LLM 통합, RAG, 에이전트 | 전무 |

**서브도메인 난립 (10개+)**

```
www.catholic.or.kr      → 메인 포털 (프레임 기반)
maria.catholic.or.kr    → 가톨릭정보 콘텐츠 서버
news.catholic.or.kr     → 가톨릭 뉴스
member.catholic.or.kr   → 회원 관리
photo.catholic.or.kr    → 갤러리
pds.catholic.or.kr      → 자료실
bbs.catholic.or.kr      → 게시판
club.catholic.or.kr     → 커뮤니티
mail.catholic.or.kr     → 이메일
m.catholic.or.kr        → 모바일 (→ maria 리다이렉트)
```

**콘텐츠 강점 (반드시 보존해야 할 자산)**

- 성경 다중 번역본 (구약/신약/NAB/200주년기념성서 비교)
- 매일미사 텍스트 + 오디오
- 성무일도, 가톨릭 성가 DB
- 23명+ 성인 DB, 성지/사적지 정보
- 전국 18개 교구 주보
- 한국가톨릭대사전
- 13개+ 게시판, 커뮤니티 클럽
- 뉴스 (비교적 현대적 반응형 디자인)

**핵심 UX 문제점**

1. 프레임 기반 구조 → URL 공유, 북마크, SEO 불가
2. 정보 과부하 → 메인 페이지에 모든 콘텐츠 압축
3. 서브도메인별 디자인 불일치
4. 오디오 플레이어가 페이지 이동 시 끊김 (SPA 아님)
5. 모바일 사이트에 메뉴 25개+ → 탐색 어려움
6. 검색 경험 부족, 접근성(WCAG) 미충족

## 3. 에이전트 시대 이전 vs 이후

### 3.1 패러다임 전환 개요

```
┌─────────────────────────────────────────────────────────────────┐
│                    2026 이전 (현재)                               │
│                                                                 │
│  [사용자] → 브라우저 → [웹사이트] → 메뉴 탐색 → 정보 발견        │
│                                                                 │
│  • 사람이 직접 웹사이트 방문                                      │
│  • 메뉴를 클릭해서 정보 탐색                                      │
│  • 검색 엔진이 키워드로 인덱싱                                    │
│  • UI/디자인이 정보 전달의 핵심                                   │
│  • 정보가 사이트 안에 갇힘 (사일로)                               │
└─────────────────────────────────────────────────────────────────┘

                          ⬇ 전환 ⬇

┌─────────────────────────────────────────────────────────────────┐
│                    2027 이후 (에이전트 시대)                      │
│                                                                 │
│  [사용자] → [개인 AI 에이전트] → API/MCP → [데이터] → 답변       │
│                                                                 │
│  • 에이전트가 사용자 대신 정보 수집                               │
│  • API로 구조화된 데이터에 직접 접근                              │
│  • 사용자는 "오늘 미사 몇 시야?" 라고 물으면 끝                   │
│  • 에이전트가 여러 소스를 조합해서 답변                            │
│  • 웹사이트는 에이전트 + 사람 모두를 위해 설계                     │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 구체적 비교: 굿뉴스의 Before & After

| 사용 시나리오 | 에이전트 이전 (현재) | 에이전트 이후 (2027) |
|-------------|-------------------|-------------------|
| "오늘 미사 독서가 뭐지?" | catholic.or.kr 접속 → 매일미사 클릭 → 해당 날짜 찾기 → 읽기 | "오늘 미사 독서 알려줘" → 에이전트가 API로 즉시 조회하여 음성/텍스트로 전달 |
| "근처 성당 미사 시간" | 전국 성당 검색 → 교구 선택 → 성당 찾기 → 미사 시간표 확인 | "가까운 성당 저녁미사 있어?" → 위치+시간 기반 자동 조회, 캘린더에 추가까지 |
| "견진성사 준비는?" | 가톨릭 교리 메뉴 → 7성사 → 견진 → 읽기 | "견진성사 준비 방법 알려줘" → 교리서+본당 프로그램+일정을 종합 안내 |
| "이번 주 헌금" | 하상앱 열기 → 가톨릭페이 → 충전 → QR 스캔 | "이번 주 헌금 5만원 보내줘" → 에이전트가 결제 처리 (AP2 프로토콜) |
| 강론 준비 (사제) | 매일미사 텍스트 + 성서해설 각각 검색 → 수동 종합 | "오늘 복음에 대한 교부들의 해설과 한국적 맥락 정리해줘" → RAG 기반 종합 자료 생성 |
| "세례 증명서 필요해" | 본당 사무실 방문 또는 전화 → 양업시스템에서 출력 | "세례 증명서 발급해줘" → 에이전트가 양업 API로 본인 확인 후 디지털 증명서 발급 |

### 3.4 핵심 프로토콜: 에이전트 시대의 인프라

2026-2027년, AI 에이전트 생태계를 연결하는 핵심 프로토콜:

| 프로토콜 | 개발사 | 역할 | 가톨릭 시스템 적용 |
|---------|-------|------|-----------------|
| **MCP** (Model Context Protocol) | Anthropic | 에이전트 ↔ 도구 연결 | 굿뉴스 데이터를 MCP 서버로 노출하여 모든 AI가 접근 |
| **A2A** (Agent-to-Agent) | Google | 에이전트 간 통신 | 본당 에이전트 ↔ 교구 에이전트 간 협업 |
| **AP2** (Agent Payment) | Google | 에이전트 자율 결제 | 교무금/헌금의 에이전트 기반 결제 |
| **llms.txt** | 커뮤니티 | AI에게 사이트 안내 | catholic.or.kr/llms.txt로 AI가 사이트 구조 파악 |

**Gartner 예측**: 2026년 기업 앱의 40%가 AI 에이전트를 탑재 (2025년 5% 미만에서 급증)
**Salesforce 조사**: 기업들이 평균 12개 에이전트를 운용 중, 2년 내 67% 증가 전망

## 4. 2027년형 굿뉴스 비전

### 4.1 핵심 원칙

1. **API-First**: 모든 콘텐츠와 데이터를 API로 노출 → 웹, 앱, 에이전트 모두가 동일 데이터 소비
2. **AI-Native**: 검색이 아닌 대화, 탐색이 아닌 추천, 수동이 아닌 자동
3. **Agent-Ready**: MCP 서버 + llms.txt + Schema.org/JSON-LD로 에이전트가 정보를 찾고 활용
4. **신자 생애주기 중심**: 세례 → 첫영성체 → 견진 → 혼인 → 임종까지 전 생애 지원
5. **통합 플랫폼**: 10개+ 서브도메인 → 단일 Next.js 앱

## 5. 양업 ERP 혁신 방안

### 5.1 현재 양업시스템의 한계

양업시스템은 2023년 "본당양업22"로 차세대 리뉴얼을 완료했고, 쿠버네티스 기반 클라우드(HyperCloud)로 전환된 상태. 시스템 자체보다 **연동과 활용**이 핵심 과제.

| 한계 | 설명 |
|------|------|
| 굿뉴스와 단절 | 양업 데이터가 굿뉴스 콘텐츠와 연동되지 않음 |
| 하상앱 보급률 0.67% | 600만 신자 중 4만명만 앱 사용 |
| 일방향 관리 | "기록과 관리" 중심, 신자 참여 부족 |
| 외부 API 미공개 | 폐쇄형 시스템, 에이전트 접근 불가 |
| AI 활용 초기 단계 | 챗봇 수준, 고도화 필요 |

## 6. On-Premise vs Cloud 보안 분석

### 6.1 현재 상황: On-Premise + Classic ASP의 보안 위험

**위험 수준: 극히 높음 (Critical)**

```
현재 상태의 위험 요소:

1. Classic ASP → 보안 패치 종료, 새 취약점 무방비
2. On-Premise → 보안 전문 인력 부족, 24/7 모니터링 불가
3. 암호화 미비 → 사회사목국 사이트는 HTTP (비암호화)
4. 감사 로그 부재 → 침해 시 추적 불가
5. 접근 제어 취약 → 인증/인가 시스템 부재
6. SQL Injection → Classic ASP의 인라인 SQL은 매우 취약
7. 재해 복구 취약 → 서버 장애 시 데이터 손실 위험
```

**실제 사례 - ASP.NET Machine Key 악용 (2025.02)**:
- Microsoft 보안 블로그에서 공개: 공개된 ASP.NET Machine Key를 악용한 대규모 공격
- 3,000개 이상의 공개된 키가 악용 가능 상태
- 아시아 지역 서버가 주요 타겟
- 레거시 ASP/IIS 서버가 주된 피해 대상

### 6.2 핵심 질문에 대한 답변

#### Q1: "클라우드에 올리면 데이터가 더 위험해지는 것 아닌가?"

**정반대입니다.**

| 보안 영역 | On-Premise (현재) | Cloud (GCP) |
|-----------|-----------------|-------------|
| 물리 보안 | 서버실 잠금장치 수준 | 24/7 무장경비, 생체인증, 비디오감시, 다중 인증 |
| 패치 관리 | 수동 (52%가 30일 내 미적용) | 인프라 레벨 자동 패치 |
| 암호화 | 별도 구축 필요 (현재 미적용 추정) | 저장(at-rest) + 전송(in-transit) **기본 제공** |
| 감사 로그 | 없거나 미비 | Cloud Audit Logs 자동 기록 |
| 침입 탐지 | 별도 솔루션 필요 | Security Command Center, Chronicle 내장 |
| DDoS 방어 | 별도 장비 필요 | Cloud Armor 기본 제공 |
| 보안 투자 | 연간 수백만~수천만원 | Google: Wiz 인수 $32B (42조원), 연간 보안 투자 수조원 |
| 전문 인력 | 전산정보실 소수 인원 | Google 보안팀 수천명 + Mandiant 침해대응팀 |
| 인증 | 없음 | ISO 27001, SOC 2, CSAP 등 다수 |

**핵심**: 전산정보실의 소규모 팀이 구축할 수 있는 보안 수준은, Google이 연간 수십조원을 투자하여 구축한 보안 인프라에 비할 수 없습니다.

#### Q2: "신자의 신앙 정보는 민감정보인데 클라우드에 올려도 되나?"

**오히려 클라우드가 법적 요구사항을 충족하기 쉽습니다.**

한국 개인정보보호법 제23조: 신앙(faith) 정보는 **민감정보**로 분류 → 가장 높은 보호 수준 필요

| 법적 요구사항 | On-Premise 충족 가능성 | Cloud 충족 가능성 |
|-------------|---------------------|----------------|
| 암호화 (저장/전송) | 별도 구축 비용+인력 필요 | 기본 내장 |
| 접근 통제 (IAM) | 수동 관리 | IAM + MFA + RBAC 내장 |
| 감사 추적 | 별도 시스템 필요 | 자동 기록 |
| 침해 대응 | 자체 대응 (인력 부족) | 24/7 SOC + 전문팀 |
| 데이터 국내 보관 | 물리적으로 보장 | **GCP 서울 리전 (asia-northeast3)** 선택 가능 |

#### Q3: "AI를 쓰면 개인정보가 학습에 사용되어 유출되지 않나?"

**Google Cloud Vertex AI / Firebase Genkit 사용 시 보장 사항:**

1. **학습 제한**: Google 서비스 약관 제17조 - 고객 데이터를 AI 모델 학습에 미사용
2. **Zero Data Retention**: 기본 인메모리 캐싱(24시간 TTL), 프로젝트 레벨에서 비활성화 가능
3. **데이터 레지던시**: 서울 리전에서 데이터 저장/처리 보장
4. **VPC 격리**: Virtual Private Cloud 내에서 AI 처리, 외부 유출 차단
5. **컴플라이언스**: SOC, ISO, HIPAA 인증 보유

**Private AI 배포 옵션:**

```
[Firebase/GCP 서울 리전]
├── Firestore (교적, 개인정보) → 암호화 저장
├── Vertex AI (AI 처리) → VPC 내 격리
│   ├── Gemini 2.5 Flash → 일반 질의 (교리, 성경)
│   └── Gemini 2.5 Pro → 복잡한 추론 (사목 분석)
├── Vector DB → 교리/문헌 임베딩 (비개인정보)
└── Cloud Functions → 비즈니스 로직

개인정보 흐름:
• 교리 질문 → 개인정보 불포함 → AI 직접 처리
• 교적 조회 → 인증 후 Firestore 직접 조회 → AI 미경유
• 사목 분석 → 익명화/집계 데이터만 AI에 전달
```

#### Q4: "On-Premise를 고수하면서 AI를 쓸 수 있나?"

**사실상 불가능에 가깝습니다.**

| 요구 사항 | On-Premise 구현 | Cloud 구현 |
|-----------|---------------|-----------|
| LLM 서빙 | GPU 서버 구매 (수억원+) + 직접 운영 | Vertex AI API 호출 (사용량 과금) |
| 벡터 DB | 직접 구축/관리 | Vertex AI Vector Search 또는 Pinecone |
| 모델 업데이트 | 수동 다운로드/배포 | 자동 업데이트 |
| 스케일링 | 서버 추가 구매 | 자동 스케일링 |
| 비용 | 초기 수억 + 월 수천만원 운영비 | 월 수십~수백만원 (사용량 기반) |

**결론**: Classic ASP + On-Premise 조합은 AI 시대의 어떤 서비스도 구현할 수 없습니다. 클라우드 전환은 선택이 아닌 필수입니다.

### 6.3 보안 전환 로드맵

```
Phase 1 (즉시): 긴급 보안 조치
├── 사회사목국 등 HTTP 사이트 → HTTPS 전환
├── 기존 ASP 서버 보안 패치 점검
└── 중요 데이터 백업 체계 수립

Phase 2 (3개월): 클라우드 기반 구축
├── GCP 프로젝트 생성 (서울 리전)
├── Firebase Auth + IAM 설정
├── VPC 네트워크 설계
├── Firestore 보안 규칙 설계
└── 데이터 마이그레이션 계획 수립

Phase 3 (6개월): 서비스 전환
├── 신규 시스템을 클라우드에서 운영
├── 기존 데이터 마이그레이션 실행
├── Zero Trust 접근 제어 적용
└── On-Premise 서버 단계적 퇴역

Phase 4 (지속): 보안 운영
├── Cloud Audit Logs 모니터링
├── Security Command Center 대시보드
├── 정기적 보안 감사
└── 개인정보 영향평가 수행
```

| **프론트엔드** | **Next.js 15** (App Router) | SSR/ISR로 SEO 최적화, React 생태계, 성능 |
| **UI 라이브러리** | **MUI v7** | 가톨릭 브랜드에 맞는 커스텀 테마, 접근성 내장 |
| **백엔드** | **Firebase** (Firestore, Auth, Functions, Storage) | 서버리스, 자동 스케일링, 보안 내장 |
| **AI** | **Gemini 2.5 Flash** (일반) + **Gemini 2.5 Pro** (추론) | Google Cloud 네이티브, 한국어 성능 우수, 비용 효율 |
| **벡터 DB** | **Pinecone** 또는 **Vertex AI Vector Search** | RAG용 임베딩 저장/검색 |
| **개발 도구** | **Claude Code** | AI 페어 프로그래밍으로 개발 속도 5-10배 향상 |
| **호스팅** | **Vercel** (프론트) + **GCP** (백엔드) | Next.js 최적 호스팅, 글로벌 CDN |
| **모니터링** | **Datadog** (기존 사용 중) | 기존 GTM + Datadog RUM 유지 |

**대안 비교**:

| 옵션 | 장점 | 단점 |
|------|------|------|
| Firebase + Next.js | 서버리스, 빠른 개발, 자동 스케일링 | Firestore 쿼리 제한 (복잡한 JOIN 불가) |
| Supabase + Next.js | PostgreSQL 기반, 복잡한 쿼리 가능 | Firebase 대비 생태계 작음 |
| AWS Amplify + Next.js | AWS 생태계 활용 | Firebase 대비 설정 복잡 |
| PocketBase + Next.js | 셀프호스팅, 간단 | 스케일링 한계, 보안 직접 관리 |

**Firebase가 최적인 이유**:
- 교적/성사 데이터는 문서(Document) 구조에 적합 → Firestore 최적
- 인증(Auth)이 내장 → OAuth, 이메일, 전화번호 로그인 즉시 지원
- 보안 규칙이 선언적 → 민감정보 접근 통제 용이
- Cloud Functions로 서버리스 백엔드 → 운영 부담 최소
- Firebase Genkit으로 AI 기능 통합 → Gemini 직접 연동
- 실시간 DB → 커뮤니티, 알림, 채팅 기능에 적합
